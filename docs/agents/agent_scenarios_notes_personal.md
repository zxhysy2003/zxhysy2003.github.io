# 什么时候需要 Agent，而不是普通 Workflow？


> 这份笔记是基于 Anthropic《Building effective agents》和 OpenAI《A practical guide to building AI agents》整理的，主要用于个人复习。重点不是背概念，而是以后做系统设计时，能判断“这个功能到底该不该上 agent”。

---

## 1. 先记住一个判断口径

我觉得最重要的一句话是：

> **如果流程能提前写死，就用 workflow；如果只能确定目标，但执行步骤要根据中间结果不断变化，才考虑 agent。**

比如：

```text
用户提交问卷
→ 保存画像
→ 根据标签和难度推荐课程
→ 返回结果
```

这种流程虽然也可以接大模型，但本质上还是 workflow。因为每一步都提前确定好了。

而 agent 更像这样：

```text
用户说一个目标
→ 系统先判断要查什么
→ 调用工具拿到结果
→ 根据结果决定下一步
→ 可能继续查，也可能改计划
→ 最后给出结果
```

所以 agent 的重点不是“用了大模型”，而是：**模型能不能在过程中自己决定下一步怎么做。**

---

## 2. Workflow 和 Agent 的区别

### Workflow

Workflow 是程序员提前设计好的流程。系统只要按照流程执行就行。

特点：

- 步骤比较固定；
- 分支有限；
- 输入通常比较结构化；
- 每一步调用什么工具基本确定；
- 出错后一般走固定异常处理；
- 便宜、稳定、容易调试。

典型例子：

```text
判断是否冷启动
→ 是：走冷启动推荐
→ 否：走协同过滤推荐
→ 查询知识图谱
→ 融合排序
→ 返回推荐结果
```

这个流程看起来挺复杂，但它仍然是 workflow，因为流程是提前设计好的。

### Agent

Agent 是程序员提供工具、规则和边界，模型在执行过程中自己决定下一步。

特点：

- 目标明确，但步骤不一定明确；
- 需要根据中间结果调整计划；
- 可能要多次调用工具；
- 工具调用顺序不固定；
- 失败后可以尝试诊断、修正、重试；
- 成本更高，延迟更高，也更难控制。

可以简单理解为：

```text
Workflow：程序员提前想好怎么做。
Agent：程序员给出工具和边界，让模型自己判断怎么做。
```

## 3. 什么时候不要急着用 Agent？

两篇文章都提醒了一点：**不要因为 agent 听起来高级就上 agent。**

很多功能其实用普通 workflow 就够了。比如：

- 登录注册；
- 课程搜索；
- 收藏课程；
- 学习进度统计；
- 冷启动问卷推荐；
- 固定 RAG 问答；
- 固定文档生成流程；
- 推荐结果融合排序。

这些功能的共同点是：流程比较明确，系统该查什么、算什么、返回什么，基本都能提前设计。

这里要注意一个误区：

> **不是用了 LLM 就是 agent。**

比如一个固定流程里有一步是“调用 LLM 总结文本”，这只是 workflow 中用了 LLM，不代表整个系统就是 agent。

---

## 4. 什么场景更适合 Agent？

我可以从下面几个角度判断。

### 4.1 只知道目标，不知道固定步骤

如果用户只说目标，系统需要自己规划步骤，就更像 agent。

例如：

> 我想两个月内学会 Spring Boot，并做一个后端项目，你根据我的情况帮我安排。

这个任务不是简单返回几门课就结束。系统可能要先查用户学过什么，再查知识点掌握情况，再找缺失先修，再生成学习路径。

可能的过程是：

```text
理解用户目标
→ 查询用户画像
→ 查询学习记录
→ 查询知识点掌握度
→ 查询课程知识图谱
→ 找缺失先修知识
→ 生成阶段性学习计划
```

不同用户的情况不同，步骤和重点也会不同，所以适合 agent。

---

### 4.2 后一步强依赖前一步结果

Agent 很典型的循环是：

```text
观察 → 思考 → 行动 → 再观察 → 再思考 → 再行动
```

比如代码修复：

```text
读取报错
→ 定位文件
→ 修改代码
→ 运行测试
→ 测试失败
→ 分析失败原因
→ 再修改
→ 再测试
```

这里不能提前写死所有步骤，因为测试结果决定下一步怎么走。

如果一个任务有明显的“看结果再决定下一步”，就可以考虑 agent。

---

### 4.3 需要动态选择工具

如果每次都固定查 MySQL、Redis、Neo4j，那是 workflow。

但如果系统要根据问题自己决定查什么，就更像 agent。

例如用户问：

> 我为什么学推荐系统总是学不懂？

系统可能需要自己判断：

```text
要不要查学习记录？
要不要查这门课的知识点？
要不要查前置知识？
要不要查用户是否跳过了基础课程？
```

这时候工具调用不是固定的，而是由模型根据当前问题和中间结果决定。

---

### 4.4 用户输入很模糊

Workflow 更喜欢结构化输入，例如：

```json
{
  "goal": "backend",
  "level": "beginner",
  "tags": ["Java", "Spring Boot"]
}
```

Agent 更适合自然语言里的模糊需求，例如：

> 我想找个适合我现在水平的项目练手，最好和毕业设计相关，但不要太难。

这句话里面其实有很多需要判断的地方：

- “现在水平”到底是什么水平？
- “毕业设计相关”是推荐系统、知识图谱、Spring Boot，还是 agent？
- “不要太难”怎么判断？
- 用户是想要课程、项目题目，还是完整学习计划？

这种输入不适合硬写很多 if-else，更适合让 agent 结合工具和上下文逐步判断。

---

### 4.5 传统规则越来越难维护

OpenAI 那篇文章里提到一个很实用的点：如果一个系统的规则越来越复杂，维护成本越来越高，就可以考虑 agent。

比如推荐规则写成这样：

```text
if 用户是初级 and 目标是后端 and 学习时长少 and 已学课程数少 ...
    推荐 A
elif 用户是中级 and 最近学习中断 and 收藏过 Spring Boot ...
    推荐 B
else if ...
```

一开始规则还好，后面规则越来越多，就会变成一堆补丁。

不过这里也不能理解成“有了 agent 就不要规则”。更合理的做法是：

```text
规则负责边界和安全；
Agent 负责复杂判断和动态执行。
```

---

### 4.6 任务需要失败恢复

如果系统失败后只是返回错误，用 workflow 也可以。

但如果希望系统继续排查原因，就更适合 agent。

例如：

> 推荐接口返回为空，帮我排查原因。

Agent 可能会这样做：

```text
查用户行为数据
→ 查推荐缓存
→ 查 Python 推荐服务
→ 查 Neo4j 图谱数据
→ 查日志
→ 判断是冷启动问题、CF 无结果，还是图谱查询失败
```

排错任务的路径往往不确定，所以 agent 比固定 workflow 更合适。

## 5. 一个简单判断表

以后可以用这个表快速判断。

| 问题 | 更偏 Workflow | 更偏 Agent |
|---|---|---|
| 步骤能不能提前写好？ | 能 | 不能完全写死 |
| 输入是不是结构化？ | 是 | 否，比较自然语言和模糊 |
| 工具调用是不是固定？ | 固定 | 需要动态选择 |
| 后一步是否依赖中间结果？ | 很少 | 经常 |
| 失败后要不要自动排查？ | 不需要 | 需要 |
| 分支数量多不多？ | 有限 | 很多，甚至无法穷举 |
| 成功标准是否清楚？ | 清楚 | 最好也要清楚 |
| 是否能提供工具和数据反馈？ | 不一定 | 必须要有 |

我自己的记忆方式是：

```text
能画成固定流程图 → workflow
只能画成循环和决策 → agent
```

## 6. 结合课程学习系统的例子

### 6.1 不太需要 Agent 的功能

下面这些功能用 workflow 更合适：

| 功能 | 原因 |
|---|---|
| 用户登录/注册 | 标准业务流程 |
| 课程搜索 | 查询条件明确 |
| 课程收藏 | 标准 CRUD |
| 学习进度统计 | 固定数据聚合 |
| 能力雷达图 | 固定指标计算 |
| 冷启动问卷推荐 | 用户填画像后按规则推荐 |
| 协同过滤推荐 | 聚合评分 → SVD → 返回结果 |
| 知识图谱缺失先修查询 | 查询语句和路径比较固定 |
| 融合推荐排序 | 公式固定，流程明确 |

比如推荐主链路：

```text
判断冷启动
→ 冷启动：按画像召回课程
→ 非冷启动：协同过滤召回
→ 查询知识图谱
→ 计算可学习性
→ 融合排序
→ 返回推荐列表
```

这个流程已经很清楚了，不需要强行改成 agent。

---

### 6.2 比较适合 Agent 的功能

更适合 agent 的地方，应该放在系统上层，比如“学习助手”或“学习顾问”。

#### 场景一：学习规划 Agent

用户说：

> 我想两个月内学会 Spring Boot，并完成一个后端项目，你根据我的基础帮我安排。

Agent 可以：

```text
查用户画像
→ 查已学课程
→ 查知识点掌握度
→ 查 Spring Boot 相关课程
→ 查缺失先修
→ 生成学习路径
→ 给出阶段性计划
```

这个功能适合 agent，因为每个用户基础不同，路径也不同。

#### 场景二：学习诊断 Agent

用户说：

> 我为什么学推荐系统总是学不懂？

Agent 可以：

```text
查最近学习记录
→ 查推荐系统课程知识点
→ 查用户掌握度
→ 查前置知识是否缺失
→ 分析可能原因
→ 推荐补学路径
```

这里不是单纯推荐课程，而是分析“为什么学不懂”。

#### 场景三：学习计划调整 Agent

用户说：

> 这周太忙了，原来的学习计划跟不上，帮我重新调整一下。

Agent 可以：

```text
读取原计划
→ 查看已完成内容
→ 找出延期部分
→ 保留关键知识点
→ 压缩非核心内容
→ 生成新计划
```

这类任务的重点是动态调整。

#### 场景四：智能排错 Agent

用户说：

> 推荐接口返回为空，帮我查一下原因。

Agent 可以：

```text
查用户行为
→ 查推荐缓存
→ 查推荐服务状态
→ 查图谱数据
→ 查日志
→ 判断根因
→ 给出修复建议
```

排错路径不固定，很适合 agent。

## 7. 如果真的要做 Agent，需要准备什么？

OpenAI 那篇文章里把 agent 拆成三个核心部分：

```text
Model + Tools + Instructions
```

我可以这样理解：

### 7.1 Model：负责判断和规划

模型负责理解用户目标，决定下一步查什么、怎么分析、什么时候停止。

但模型不能凭空知道用户数据，所以必须依赖工具。

### 7.2 Tools：给 Agent 可调用的能力

课程系统里可以提供这些工具：

```text
getUserProfile(userId)
getLearningBehaviors(userId)
getUserMastery(userId)
searchCourses(keyword, difficulty, tags)
getCourseDetail(courseId)
getCourseKnowledgePoints(courseId)
getMissingPrerequisites(userId, courseId)
generateLearningPath(userId, goal)
saveLearningPlan(userId, plan)
```

工具设计要清楚，返回结果也要结构化。否则模型很难稳定使用。

### 7.3 Instructions：告诉 Agent 不能乱来

需要给 agent 明确规则，例如：

```text
你是课程学习顾问。
不能编造用户学习记录。
涉及用户数据时必须调用工具查询。
推荐课程时要说明理由。
数据不足时要说明不确定。
不能直接保存学习计划，除非用户确认。
高风险操作必须请求确认。
最多调用 N 次工具，避免无限循环。
```

这里其实就是 guardrails。没有这些边界，agent 很容易失控或输出不可靠内容。

## 8. 一个最小可实现版本

如果要在毕业设计里加 agent，可以先做一个比较小的版本，不要一开始做太大。

### 功能名

学习顾问 Agent

### 用户入口

```text
我想学 Spring Boot，但 Java 基础一般，应该怎么学？
```

### 可用工具

```text
1. 查询用户画像
2. 查询学习行为
3. 查询知识点掌握度
4. 查询课程知识图谱
5. 查询课程列表
```

### 输出内容

```text
1. 当前基础判断
2. 缺失知识点
3. 推荐学习路径
4. 推荐课程
5. 每一步的理由
6. 后续学习建议
```

### 为什么这个版本适合 Agent？

因为它满足几个条件：

- 用户输入是自然语言；
- 每个用户基础不同；
- 需要查多个数据源；
- 需要根据查询结果调整推荐路径；
- 输出不是简单列表，而是个性化分析。

这个版本范围比较可控，也比较容易和现有系统结合。

---

## 9. 最后记住这几句话

1. **能用固定流程解决的，不要上 agent。**
2. **Agent 适合目标明确但路径不确定的任务。**
3. **Agent 的价值在于动态决策、工具选择、反馈循环和失败恢复。**
4. **Agent 的难点不只是 prompt，而是工具、权限、边界、日志、停止条件和评估。**
5. **对于课程系统，推荐主链路更像 workflow；学习顾问、学习诊断、计划调整、智能排错更像 agent。**
